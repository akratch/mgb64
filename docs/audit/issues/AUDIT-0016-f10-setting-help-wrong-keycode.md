# AUDIT-0016: FPS-Toggle Help Publishes the Wrong SDL Keycode for F10

| Field | Value |
| --- | --- |
| Status | Fixed |
| Severity | S4 - incorrect player-facing configuration guidance |
| Priority | P3 |
| Area | Input settings / documentation |
| Evidence level | Runtime reproduced and source proven |
| Confidence | High |
| Origin | Newly confirmed by this audit |
| Affected configurations | Settings UI, generated `ge007.ini`, `--list-settings`, and `docs/ENV_FLAGS.md` |

## Summary

The `Input.FpsToggleKey` help text says that F10's SDL keycode is `1073741899`.
The compiled `SDLK_F10` value registered and consumed by the application is
`1073741891`. A player following the numeric guidance writes a different key
and loses the advertised F10 binding.

The implementation default is correct; only the duplicated numeric
documentation is wrong.

## Evidence

[`platformRegisterConfig`](../../../src/platform/platform_sdl.c#L2243)
registers the setting with `SDLK_F10`, then hardcodes `1073741899` in the help
string at line 2247. The actual global default is also `SDLK_F10` at line 1297.

[`ui_overlay.cpp`](../../../src/app/ui_overlay.cpp#L114) compares keyboard
events to the setting value, falling back to `SDLK_F10`. It therefore consumes
the correct compiled constant, not the number in the help string.

A ROM-free clean-config dump generated by the current executable writes:

```ini
# SDL keycode that toggles the FPS overlay without opening the menu
# (default F10 = 1073741899).
FpsToggleKey=1073741891
```

[`ENV_FLAGS.md`](../../ENV_FLAGS.md) repeats the wrong `1073741899` value because
its description is generated from the same registered help text.

## Reproduction

No ROM is required. Use an isolated savedir or a read-only faithful session:

```sh
build/ge007 --faithful --dump-config | \
  sed -n '/FPS key/,+5p'
```

The help comment and effective value disagree by eight. Setting
`Input.FpsToggleKey=1073741899` and pressing F10 is a negative control: F10 no
longer matches the configured integer, whereas `1073741891` does.

## Root Cause

The numeric value was copied into free-form help instead of being formatted
from `SDLK_F10` or omitted. SDL keycodes are not memorable interface constants,
so the duplicated literal drifted from the compiled value at introduction.
The environment-flag documentation generator faithfully propagated the bad
description.

## Required End State

Remove the hardcoded decimal from player-facing prose unless numeric editing is
an intentionally supported interface. If it remains, generate it from
`SDLK_F10` in the settings-list/dump formatter and from the same schema data in
documentation.

Prefer displaying a key name such as `F10` in the launcher and overlay, while
serializing the platform-independent canonical binding representation selected
by the input-rebinding design. Raw SDL integers may remain an advanced fallback
but must be exact and round-trip safely.

## Acceptance Criteria

- The effective default and every displayed/documented numeric value agree.
- Pressing F10 toggles the FPS overlay with an untouched configuration.
- Saving and reloading the default binding preserves F10.
- The settings UI displays `F10`, not a misleading raw integer.
- `--list-settings`, `--dump-config`, generated `ge007.ini`, and
  `docs/ENV_FLAGS.md` do not contradict one another.
- A unit test compares the registered default to `SDLK_F10` and verifies the
  serialized value's round trip.
- Changing a binding to another key and restoring the default continues to work
  without requiring a hand-maintained decimal.

## Verification Plan

Extend the ROM-free config-schema test to locate `Input.FpsToggleKey`, assert
its default equals `SDLK_F10`, serialize and reload it, and compare any rendered
numeric help token to that value. Add an app event test that sends F10 at the
default and verifies the overlay toggle, then sends it after a rebind and
verifies it no longer toggles. Regenerate environment documentation and assert
there is no stale `1073741899` occurrence.

## Related Work

- The post-boot UX plan correctly specifies `SDLK_F10`; the defect is limited to
  the decimal annotation added during implementation.
- [AUDIT-0011](AUDIT-0011-malformed-numeric-settings.md) separately covers
  validation of hand-entered numeric setting values.

## Resolution <!-- triage-2026-07-14 -->

Verified already fixed in commit `6c1a23d` (ledger Status was stale). The F10 help now advertises the correct SDL keycode 1073741891 (platform_sdl.c:2269).
