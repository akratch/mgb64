# FID-0037 / FID-0039 / FID-0107 — converter + Spectrum-renderer batch

Three small, self-contained port findings in the asset converters and the
ZX-Spectrum easter-egg DL builder. None touches sim determinism, the sim-state
hash, or any struct layout. Authority order: **retail ASM > ares > decomp C**;
the `#else` reference bodies are not authority.

---

## FID-0107 — Spectrum-screen SETTILE emits tile 0, not tile 7 (port-defect, FIXED)

**Class:** port-defect · **Surface:** renderer · **Status:** landed
(ROM-free `spectrum_settile` lane; fail-on-revert proven).

### Retail behaviour (instruction-level)

`spectrum_draw_screen` (retail US ASM `glabel 7F0D33B4`, transcribed in
`src/game/spectrum.c`) hand-builds the emulator screen DL. It loads the SETTILE
low word (w1) once into `$s1` and stores it at both SETTILE sites:

```
/* 107F50 7F0D3420 3C110700 */  lui   $s1, 0x700          # $s1 = 0x07000000
...
/* 107F5C 7F0D342C ACB10004 */  sw    $s1, 4($a1)          # palette SETTILE (w0=0xF5000300) w1
...
/* 1080B8 7F0D3588 ACB10004 */  sw    $s1, 4($a1)          # per-tile SETTILE (w0=0xF5100000) w1, in the 3x4 loop
```

`0x07000000` sets the SETTILE **tile** field (w1 bits 26..24) to **7 =
G_TX_LOADTILE**. Both loads that follow each SETTILE name tile 7:

- palette LOADTLUT: `0xF0000000 / 0x0703C000` (tile field 7)
- per-tile LOADBLOCK: `0xF3000000 / 0x073FF200` (tile field 7)

So the SETTILE configures the very tile the following LOADTLUT/LOADBLOCK load
into.

### Port divergence

The `NONMATCHING` C body wrote the digit-shifted constant `0x00070000` at both
sites (`spectrum.c` palette + per-tile SETTILE). Decoded, `0x00070000` is
tile = 0, plus a stray **cmt = 1** (bit 18) and **maskt = 12** (bits 17..14):
the SETTILE reconfigures tile 0 while the loads still target tile 7. Observable
iff the port's Fast3D binds LOADTLUT/LOADBLOCK to a per-tile descriptor rather
than the immediately-preceding SETTILE (the tile-7 descriptor is then stale for
the Spectrum screen). Same wrong-load-tile-in-GBI class as FID-0104.

### Fix

`spectrumSettileLoadTileW1(legacy)` (`src/platform/spectrum_settile.c`) returns
`0x07000000` faithful / `0x00070000` legacy; `spectrum_draw_screen` uses it at
both sites. Default faithful; `GE007_NO_SPECTRUM_LOADTILE_FIX` restores the
port constant (byte-identical to the old DL). Rest of `spectrum_draw_screen`
verified byte-identical to the ASM.

## FID-0037 — setup padnames / boundpadnames left NULL (port-defect, FIXED)

**Class:** port-defect · **Surface:** converter · **Status:** landed
(ROM-free `setup_pnames` lane; fail-on-revert proven).

### Retail behaviour

The stage setup file opens with ten big-endian 32-bit table offsets. Words 8 and
9 are `padnames` and `boundpadnames`: arrays of big-endian 32-bit file-local
offsets, terminated by a zero entry, each non-zero offset pointing (relative to
the file base) at a NUL-terminated pad-name string. Retail resolves the header
table then relocates each entry — see the decomp reference in the same file
(`prop.c` `padnames`/`boundpadnames`: resolve at the `if (local_stage->padnames)`
block, relocate at the `for (...; padnames[i].p; ...) padnames[i].p =
TRANSFORM4(...)` loop).

### Port divergence

The 64-bit `NATIVE_PORT` converter `proplvreset2` left both tables permanently
NULL (`g_CurrentSetup.padnames = NULL; /* TODO: byte-swap pname structs */`,
`prop.c:2538` in the pre-fix tree). `struct pname` is 4 bytes on N64 but the host
union `{char*; s32}` is 8, so — like the pads/boundpads — the table cannot be
cast in place; it must be re-laid-out, which is why it was stubbed.

**Player-visible effect:** small. The only consumers,
`initpathtablelinks.c` (four `if (g_CurrentSetup.padnames != NULL)` /
`boundpadnames` sites), have their bodies removed in the shipping build (release
strips the pad-name diagnostics), and the surrounding `var_s6 = 1` runs
regardless. So the tables feed only (stripped) debug name printing; nulling them
does not change gameplay or sim. The fix restores the faithful setup structure
and re-arms any future diagnostic use.

### Fix

Pure ROM-free helper `setup_pnames.c` does the byte-swap + count + base+offset
relocation; `proplvreset2`'s `build_pc_pnames` re-lays-out each table into a
host-native, NULL-terminated `struct pname` array (freed on stage reload like the
other `s_pc_*` buffers). Default faithful; `GE007_NO_PADNAMES_FIX` restores the
port defect (both NULL) — byte-identical to the old behaviour. No sim state is
touched.

## FID-0039 — OP11 "rendering unimplemented" is faithful to retail (DOCUMENTED, no fix)

**Class:** port-defect (reclassified → **documented**) · **Surface:** converter ·
**Status:** documented.

The seed finding reads `model_convert.c:881` "OP11 rendering unimplemented (field
width note)": the converter stores `0` into `Op11.unk0c[15]` because a 64-bit
pointer will not fit in `u32[16]`, with a "TODO: when OP11 rendering is
implemented" note. Investigated against the ASM/consumer graph, this is **not a
live divergence**:

1. **OP11 render is a no-op in the shipping game, not just the port.** The OP11
   dispatch (`model.c:12085` and `12155`) calls `sub_GAME_7F0737FC`, whose body
   is `return;` (`model.c:10845`) — matching its sibling `sub_GAME_7F0737EC`.
   The rodata struct is documented "Opcode 11 / unused but referenced"
   (`bondtypes.h`). OP11 nodes therefore emit **no** GBI; there is nothing to
   "implement."

2. **`unk0c[15]` is read as a pointer in exactly two places, neither observable
   on the port.** (a) The no-op render above. (b) The retail promote path
   `modelPromoteNodeOffsetsToPointers` (`model.c:14040` `PROMOTE(unk0c[15])` +
   `BaseAddr = fileramaddr`) — which uses `PROMOTE(var) = (u32)var + diff`
   (`model.c:13896`), a 32-bit-truncating fixup that cannot run on 64-bit hosts.
   The converter **replaces** that whole path (`modelConvertN64Binary` is called
   "in place of sub_GAME_7F075A90"), so it never executes on the port.

3. **The OP11 fields the port actually uses are converted correctly.**
   `BoundingVolumeRadius` (culling radius, `model.c:2763`) and `RwDataIndex`
   (`model.c:2390` / rw-slot sizing `model.c:14205`) are read from the record;
   both are populated (`model_convert.c` OP11 rodata case). Only `unk0c[15]` /
   `BaseAddr` — the pointer never dereferenced observably — are affected.

**Conclusion:** the port faithfully reproduces retail's no-op OP11 render, and
storing `NULL` in `unk0c[15]` is behaviourally invisible **and** the safe
representation (a truncated 32-bit pointer would be strictly worse if OP11 render
were ever un-no-op'd). No fix is warranted; the misleading "TODO: implement"
comment was replaced with this rationale. If a future change gives OP11 a real
render body, this finding's retest condition re-opens it: the field must then be
widened / side-pointered before `unk0c[15]` is dereferenced.
