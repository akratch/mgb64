## 1. Strategy & Honest Ceiling

Add a second `GfxRenderingAPI` implementation ‚Äî `gfx_metal_api` ‚Äî alongside the existing `gfx_opengl_api`, selected behind a macOS-only opt-in flag at the single backend seam (`gfx_pc.c:22879`), leaving OpenGL as the default everywhere and byte-identical on Linux/Windows. The DL interpreter and all T&L live in `gfx_pc.c`; the backend is reached almost entirely through the `gfx_rapi->*` vtable. The payoff is that native Metal renders the remaster's SSAO/depth-sampling path directly (`Depth32Float` + `ShaderRead`) instead of through Apple's deprecated GL-over-Metal translator, whose op-level hang on depth-texture sampling is the thing currently blocking Phase-1 lighting.

**On gameplay-invariance ‚Äî corrected.** This is a render-only, opt-in change on the existing rails, but invariance is **not** "by construction." The CPU clipper's triangle-admission math reads a GL-owned global: `GFX_CLIP_Z_SCALE = (g_depth_clamp_enabled ? 1.0f : 0.3f)` (`gfx_pc.c:223`), and `g_depth_clamp_enabled` is set exactly once, inside `gfx_opengl_init()` (`gfx_pc.c:219, 2413-2416`). The near/far clip-reject bits derive from it (`gfx_pc.c:2918-2925`). On the Metal path `gfx_opengl_init()` never runs, so unless Metal init **also** sets `g_depth_clamp_enabled = true`, the scale silently reverts to `0.3`, admitting/rejecting a different set of near-frustum triangles ‚Üí different `rooms_drawn`/`tris` bookkeeping ‚Üí different arena bytes. Invariance therefore holds **by a required, forgettable init side-effect**, and `g_depth_clamp_enabled` is a **third cross-backend coupling** (¬ß2.1) that must be hoisted out of GL ownership and set explicitly by the Metal init. This is verified in ¬ß0 / Phase-1 spike before any shader work.

**Honest ceiling:** this is a **weeks-to-months** effort (realistically ~8‚Äì12 engineer-weeks to full parity including the remaster chain), not a drop-in. libultraship (LUS, Kenix3) ships the only Fast3D-lineage Metal backend and is a genuine template, but it targets a *different, richer* interface (C++ virtual class, ~45 methods, explicit framebuffer suite, and it assumes Ship's config/threading/`spdlog` logging, its own Fast3D struct layout, and an ImGui-Metal interleave) ‚Äî so it is an **adapt-as-template**, not a lift-verbatim, and its `start_frame`/`end_frame`/present are explicitly *not* reusable (ImGui-coupled). Two areas have **no upstream template at all** (the port's RDP-memory/coverage-stencil blend modes + the per-batch XLU framebuffer snapshot, and the bespoke remaster post-FX shader) and must be written fresh. The ceiling is GL *visual parity within tolerance* plus the SSAO win ‚Äî not a PBR/materials upgrade; the engine still has no normals.

## 0. Pre-Flight Gate ‚Äî ‚úÖ TESTED 2026-07-01 ‚Üí RULED OUT (native Metal confirmed)

The entire stated payoff (¬ß5, Phase 5) is "SSAO without the `Depth32Float`+`ShaderRead` op-hang." Before committing to a months-long backend, **test and reject the cheap GL-only dodge**: render the SSAO depth pre-pass into a **color-encoded** target (`R32F` or packed `RGBA8`) and sample *that* as an ordinary color texture, sidestepping the specific depth-texture-sampling op that hangs the deprecated translator ‚Äî while staying on GL everywhere.

> **RESULT ‚Äî ¬ß0 is dead; native Metal is required.** Ran the definitive isolation: the reconstruction AO (proj-divide + cross/normalize) with its source texture bound to a **COLOR** texture instead of the depth texture (`GE007_SSAO_COLORDEPTH`, identical math). It **hangs identically to reading depth** (exit 137, ~45s force-kill). Therefore the trigger is **NOT** the depth-texture-sample op (a color `sampler2D` hangs too) ‚Äî it is the **reconstruction math itself**, regardless of texture format. The color-encode dodge cannot help, and (corollary) neither can a linZ-only linear-depth prepass, since it still needs the per-sample `ndc/projX`,`ndc/projY` divide. This *confirms* the native Metal backend rather than mooting it.

- **Deliverable:** ‚úÖ isolation spike run (reverted; tree at P0).
- **Decision:** ‚úÖ hang persists on the color source ‚Üí native backend justified ‚Üí proceed to Phase 1.
- **Effort:** ~1hr (done). **Value:** HIGH ‚Äî converted "should we spend weeks?" into a definitive yes.

*(The other historical stopgap ‚Äî keep patching GL-over-Metal / disable SSAO on macOS ‚Äî is not a fix: it's an op-level driver bug on a deprecated path Apple can remove, and the prior `textureLod` fix already failed. The color-encoded dodge above is the *good* cheap option and must be ruled out on its own merits.)*

## 2. Architecture

### 2.1 The backend seam (contract + the couplings the frontend does NOT abstract)

- Interface: `struct GfxRenderingAPI` ‚Äî 22 function pointers, `gfx_rendering_api.h:25-49`. GL implements it positionally as `gfx_opengl_api` at `gfx_opengl.c:3629-3652`.
- Selection: one line, `gfx_pc.c:22879` (`gfx_rapi = &gfx_opengl_api; gfx_rapi->init();`), inside `gfx_init()`. Metal = add `extern struct GfxRenderingAPI gfx_metal_api;` and branch: `gfx_rapi = use_metal ? &gfx_metal_api : &gfx_opengl_api;`, `#ifdef __APPLE__`-guarded, driven by a new env/config selector (`GE007_RENDERER=metal|gl` / `Video.Backend`) ‚Äî **no such selector exists today; it must be added,** and it must be read **before** `SDL_CreateWindow` (GL vs Metal window flags are mutually exclusive).
- **Non-vtable couplings hardwired to GL that Metal must dispatch per-backend (three, not two):**
  1. `set_clear_color` (`gfx_pc.c:23165`)
  2. `max_offscreen_dim` (`gfx_pc.c:4434`)
  3. **`g_depth_clamp_enabled`** (`gfx_pc.c:219, 223, 2413-2416, 2918-2925`) ‚Äî Metal init must set it `true` (consistent with dropping the `z*=0.3` hack). This is the invariance-critical one.
  - Cleanest for (1)+(2) is +2 vtable pointers ‚Üí 24-entry vtable; (3) is hoisted to a shared init responsibility.
- **The frontend is not fully backend-agnostic.** `gfx_pc.c` `#include`s `<OpenGL/gl3.h>`/`<glad/glad.h>` (`:75-78`) and makes **direct GL calls outside the vtable** that Metal must intercept or stub:
  - `glReadPixels` √ó2 (`:22991`, `:23190`) ‚Äî covered (Phase 4).
  - **`glGetTexImage` (`:21085`, under `GE007_VERIFY_GPU`)** ‚Äî a *texture* readback (not framebuffer), no vtable entry, no metal-cpp analog. Needs its own blit-readback path **or** the `GE007_VERIFY_GPU` knob is stubbed on Metal. **(Previously missed.)**
  - `SDL_GL_SwapWindow` (`:23921`), `SDL_GL_GetDrawableSize` (`:4417`) ‚Äî covered.
- **The one semantic fork the frontend already abstracts:** `z_is_from_0_to_1`. GL returns false (`gfx_opengl.c:579`); Metal returns **true**, and `gfx_pc.c:18385-18387` already remaps clip z via `z=(z+w)/2` for `[0,1]`-NDC backends (the proven Fast3D D3D path). With `g_depth_clamp_enabled=true` set by Metal init, the **CPU vertex/clip pipeline stays byte-for-byte identical between GL and Metal**, and we can drop GL's `gl_Position.z *= 0.3` spreading hack (`gfx_opengl.c:926-934`) in favor of native depth-clamp (`setDepthClipMode:Clamp`).

### 2.2 `gfx_metal.cpp` ‚Äî the backend

A single translation unit (pure C++ via header-only metal-cpp ‚Äî no Obj-C needed except SDL layer bridging) exposing `extern "C" struct GfxRenderingAPI gfx_metal_api = {‚Ä¶}`. It owns one `MTL::Device`, a primary `MTL::CommandQueue` + a dedicated readback queue, a triple/ring-buffered vertex `MTL::Buffer` pool, and ‚Äî the core architectural shift from GL immediate-mode ‚Äî **immutable-state caches**, because Metal bakes blend factors, write-mask, alpha-to-coverage, color pixel-format, MSAA sample count, *and* the vertex descriptor into the pipeline object:

- **`MTLRenderPipelineState` cache** keyed on **`(shader_id0, shader_id1, blend_mode, sample_count, color_format, colorWriteMask)`**. The `colorWriteMask` component is **not optional**: `glColorMask` is toggled dynamically for color-write-off passes (`gfx_opengl.c:1701, 2015, 2025`), and Metal bakes the mask into the PSO ‚Äî omit it and color-masked passes render with a stale (writing) pipeline. This is why the GL model (independent dynamic `load_shader`/`set_blend_mode`/`set_depth_mode` calls, `gfx_opengl.c:714/1556/1619`) collapses into *deferred state recorded by the setters and resolved at `draw_triangles`*.
- **`MTLDepthStencilState` cache** keyed on `(test, update, compare-func, zmode)`, mapping the 4 N64 zmodes; ZMODE_DEC polygon-offset ‚Üí `setDepthBias:slopeScale:clamp:` per draw. Confirm this key fully captures standalone `glDepthMask` toggles (`gfx_opengl.c:3364, 3463`).
- **`MTLSamplerState` cache** keyed on `(linear, cms, cmt)` incl. anisotropy (`gfx_opengl.c:1529-1551`). *Note:* mipmaps are deliberately disabled for NPOT (`gfx_opengl.c:1531-1536`), so anisotropy is largely moot ‚Äî do not over-engineer this cache.
- **Per-shader `MTLVertexDescriptor`** built from the combiner's dynamic float layout (the `num_floats` stride assembled today in `gfx_opengl_vertex_array_set_attribs`, `gfx_opengl.c:583-592`).
- **Rasterizer fill mode is a knob, not a shader trait:** `GE007_WIREFRAME` uses `glPolygonMode(GL_FRONT_AND_BACK, GL_LINE)` (`gfx_opengl.c:3569`) ‚Üí Metal `setTriangleFillMode:Lines` on the encoder. (Knobs are spread across rasterizer/readback/present, not only the shader generator.)

`draw_triangles` (mirrors `gfx_opengl.c:2036`) writes `gfx_pc.c`'s interleaved `buf_vbo[]` (`gfx_pc.c:18388+`, stride = num_floats ‚Äî *same layout feeds both backends unchanged*) into the ring buffer, then flushes all deferred state into the encoder: resolve PSO, bind vertex buffer + textures + sampler + depth-stencil state + depth-bias, `drawPrimitives(triangle)`.

### 2.3 The combiner ‚Üí MSL translator (the make-or-break)

There is **no shared SPIR-V/IR** in this lineage: both GL and Metal do **per-combiner string templating** from the same `struct CCFeatures` (`gfx_cc.h:77-104`, decoded by `gfx_cc_get_features` from a 64-bit `shader_id0` + ~29 `shader_id1` flag bits). The GL generator is `gfx_opengl_create_and_load_new_shader` (`gfx_opengl.c:819-1449`, ~630 lines) built on two primitives ‚Äî `shader_item_to_str` (`:731-790`) and `append_formula` (`:792-817`). LUS's `p_shader_item_to_str`/`p_append_formula` (`gfx_metal_shader.cpp:30/114`) are **structurally identical** ‚Äî same SHADER_0/INPUT_n/TEXEL0/COMBINED/NOISE cases, same single/multiply/mix/interp branches ‚Äî differing only in vocabulary (`vec4`‚Üí`float4`, `.rgb`‚Üí`.xyz`, `texture(t,uv)`‚Üí`t.sample(smp,uv)`, `gl_FragCoord`‚Üí`in.position`, `noperspective`‚Üí`[[center_no_perspective]]`, `dFdx`‚Üí`dfdx`). Strategy: **retarget THIS port's own string builder to emit MSL**, using LUS's item table as a 1:1 correctness oracle ‚Äî *not* adopt LUS's prism/O2R template engine (which drags the whole Ship runtime).

**The trap that makes this weeks:** this fork's generator has diverged far past stock Fast3D and emits port-unique features LUS's template does not contain ‚Äî N64 tile-mask wrap (`:993-1006`), the derivative-based N64 3-point texture filter (`dFdx/dFdy` + `textureLod`, `:1008-1040`), shader-side UV clamp variants (`:1079-1107`), `noperspective` at 21 sites, and ‚Äî highest-risk ‚Äî the in-shader RDP-memory/coverage blend that **samples a framebuffer texture** mid-fragment (`:1059-1075, 1238-1277`). Each must be ported to MSL and A/B-verified against GL. On knob count: the plan will **not** try to reproduce all `GE007_` behaviors. There are **~298 `GE007_` tokens in `fast3d/*.c`** (not 134); most are A/B escape hatches for already-fixed bugs and need no Metal parity. Phase 2/3 will enumerate and wire **only** the subset that gates generator/rasterizer/readback output (e.g. `GE007_WIREFRAME`, the N64-filter and RDP-blend toggles); the rest are stubbed or hard-defaulted on Metal, listed explicitly in the deliverable.

### 2.4 Window / present / readback layer

- **Window fork** (`platform_sdl.c:2090-2166`): today hardwired to `SDL_WINDOW_OPENGL` + `SDL_GL_CreateContext` (Core 4.1 forward-compat ‚Äî the deprecated path that hangs). Metal path: create the window *without* `SDL_WINDOW_OPENGL`, then `SDL_Metal_CreateView` + `SDL_Metal_GetLayer` ‚Üí `CAMetalLayer`. SDL returns `void*`, so SDL need not be built with Metal headers. HiDPI: `SDL_GL_GetDrawableSize` (`platform_sdl.c:600`) ‚Üí `layer.drawableSize`. Backend must be chosen before `SDL_CreateWindow`.
- **Present** (`gfx_pc.c:23921`): `SDL_GL_SwapWindow` ‚Üí `[cmdbuf presentDrawable:nextDrawable]; commit`. Site made backend-aware. The macOS vsync/focus-loss swap hangs (`platform_sdl.c:2211-2213, 1425-1439`) largely evaporate under Metal.
- **Frame model**: GL immediate calls ‚Üí per-frame `MTL::CommandBuffer` + `MTLRenderCommandEncoder` whose lifetime spans all `draw_triangles`; clears via `MTLRenderPassDescriptor loadAction=Clear` (not a clear call), wrapped in `NS::AutoreleasePool`. `start_frame`/`end_frame` mirror `gfx_opengl.c:3543/3621`. **ImGui is a non-issue** ‚Äî zero ImGui/nuklear anywhere in the tree ‚Äî so budget nothing for it, but the corollary is that LUS's ImGui-interleaved `start_frame`/`end_frame`/present **cannot be lifted**; ours are authored fresh.
- **Readback** (`read_framebuffer_rgb`, `gfx_opengl.c:2123`): Metal has no synchronous `glReadPixels`. Use blit-encoder ‚Üí shared `MTL::Buffer` ‚Üí `getBytes` with command-buffer `waitUntilCompleted` on the **offscreen composite** (not the drawable), plus a Y-origin flip (GL bottom-up vs Metal top-left; screenshot flip loop `gfx_pc.c:23195`). This is load-bearing: 14 call sites (`gfx_pc.c:6528‚Ä¶8036`) drive the render-classification probes the *entire validation harness* rides on. **Corrected:** these probes are diagnostic trace loggers (`gfx_trace_*`, `gfx_*_pixel_probe_*`) that `fprintf` JSON and do **not** feed sim/aim ‚Äî so there is no gameplay-coupling hazard. However, several perform **predicted-vs-actual GPU-blend checks** ‚Äî `gfx_predict_alpha_blend_channel` = `(src*a + dst*(255-a) + 127)/255` and `gfx_predict_rdp_force_blend_channel` uses N64 `>>3`/`>>5` integer math (`gfx_pc.c:7882-7900`) ‚Äî calibrated to GL's UNORM blend rounding. Metal's `RGBA8Unorm` blend can differ ¬±1 LSB, so these probes will log spurious "mismatch" on correct Metal frames unless we either (a) give them a ¬±1-LSB tolerance, or (b) reproduce the exact `>>3`/`>>5` integer arithmetic in the MSL RDP-memory blend so predicted==actual holds. **(b) makes bit-exact RDP-blend math a HARD requirement, not a "within tolerance" nicety.**

### 2.5 Remaster targets & post-FX (the second, separate MSL program)

Separate from the combiner: the scene render target (`gfx_opengl.c:648-666, 2280-2443` ‚Äî color RGBA8 + a **sampleable** depth texture, the op that hangs the GL translator), MSAA resolve (`:3600-3611`), and the ~150-line output-VI-filter GLSL fragment shader (`gfx_opengl.c:2937-3091`: FXAA, CAS sharpen, SSAO, bloom, color-grade, filmic tonemap, gamma, vignette, Bayer dither, RGB555 quantize) driven by the ping-pong `apply_output_vi_filter` (`:3246-3473`). In Metal: color = `RGBA8Unorm`, depth = `Depth32Float` w/ `RenderTarget|ShaderRead`, MSAA = `Type2DMultisample` + `storeAction=MultisampleResolve` (natively easier than GL's blit ‚Äî and **Metal can resolve MSAA depth**, so the "SSAO disabled under MSAA" limitation at `gfx_opengl.c:2255-2265` disappears; note this makes Metal's MSAA-depth behavior *diverge* from GL and it must be validated as its own case). Critical structural change: the GL filter reads/writes the **default framebuffer**; Metal has none, so the scene renders into an offscreen "screen" color texture, filter passes run texture‚Üítexture, and the final result is blitted into the `CAMetalDrawable`.

### 2.6 What is LIFTED vs ADAPTED vs FRESH

| Bucket | Items |
|---|---|
| **LIFTED** (copy the idiom, ~mechanical) | metal-cpp FetchContent recipe (from LUS `mac.cmake`, single-header-metal-cpp); device/`CAMetalLayer`/command-queue bring-up; per-shader MSL library + `RenderPipelineState` build; `MTLVertexDescriptor` from float layout; texture `replaceRegion` upload; sampler/depth-stencil caches; vertex-buffer pool; per-frame `NS::AutoreleasePool`; deferred blit-readback pattern; `depthKernel`/`convertToRGB5A1` compute kernels. **Explicitly NOT lifted:** LUS `start_frame`/`end_frame`/present (ImGui-coupled) and its `spdlog`/config/threading plumbing. |

Using LUS as a "1:1 oracle" and copying idioms produces a derivative work, so `gfx_metal.cpp`'s header **must** reproduce the LUS MIT copyright/NOTICE (mirroring the existing Emill attribution) and the CMake/vendoring must carry metal-cpp's Apache-2.0 header + NOTICE.

## 3. Phases (dependency order)

**Rails / gate vocabulary referenced in every ACCEPTANCE ‚Äî corrected:**
- **Cross-backend gameplay invariance** is proven with **`tools/compare_state.py`** on a shared RAMROM replay run on both the GL build and the Metal build (gameplay-field trace equality). It is **not** proven by `sim_invariance_gate.sh`, which only toggles `Video.RemasterFX/Ssao` OFF-vs-ON *within a single backend* and never crosses the GL/Metal boundary; and it is **not** proven by the whole-arena `sim_state_hash`, whose own header NOTE warns it folds render bookkeeping (culling counts) into the hash and will "correctly but unhelpfully" flag gameplay-neutral render differences ‚Äî exactly what a clip-order or rasterizer-order delta between backends produces. **`sim_state_hash` / `sim_invariance_gate.sh` are retained only for within-backend FX-leak checks.**
- **Screenshot parity** uses `tools/compare_screenshots.py` via `tools/renderer_parity_capture.sh` (cross-renderer capture), but **`DIFF_THRESHOLD=5` is the per-pixel change-detection threshold, not the pass tolerance.** Pass/fail is `changed_pct > --max-changed-pct`, and **`--max-changed-pct` has no default** (omit it and the tool never fails). Therefore parity acceptance requires: (i) **per-golden `--max-changed-pct` budgets defined empirically**; (ii) the faithful base path validated with **Bayer-dither + RGB555-quantize OFF** (a half-LSB rounding difference flips the dither/quantize pattern across the *entire* frame ‚Äî pathological for pixel-diff), or a compare mode that quantizes both sides to the RGB555 grid before diffing; (iii) **gross parity judged by the presentation / `active_bbox` (geometry-placement) metrics**, with raw pixel-diff reserved for masked ROIs.
- **ASan clean**; **zero warnings** (note `-Werror=deprecated-declarations` under `APPLE+PORT_STRICT`, `CMakeLists.txt:480-482`); **CI green** ‚Äî all Metal work is `if(APPLE)`-guarded + runtime-opt-in, so Linux/Windows/GL CI is untouched and byte-identical.

## 4. Total Effort, Biggest Risks, Go/No-Go

**Realistic total:** **~8‚Äì12 engineer-weeks** for one engineer to full parity + remaster + SSAO. A **faithful-base-path MVP** (Phases 1‚Äì3, no remaster) is **~5‚Äì7 weeks ‚Äî treat 6 weeks as the floor, not the expected value,** because Phase 2 is the explicit 70%-risk gate and the RDP/XLU encoder-break path lands in Phase 3. These are wall-clock working estimates dominated by *validation*, not initial coding ‚Äî budget the majority of Phases 2‚Äì4 for A/B parity loops across 20 levels + split-screen and the enumerated diag knobs, not first-render.

**Biggest risks (re-ranked):**
1. **RDP-memory / coverage-stencil blend + per-batch XLU snapshot** ‚Äî mgb64-proprietary, zero external reference, load-bearing for glass/jungle. In Metal you cannot generally sample the color attachment you are writing without either programmable blending (`[[color(0)]]` inout, tile-only, Apple-GPU-specific) or a copy-to-sampled-texture per batch, so this **forces an encoder break per snapshot** ‚Äî a structural per-batch cost *and* correctness hazard ‚Äî and it demands **bit-exact `>>3`/`>>5` integer math** to keep the readback probes honest. Spiked in Phase 1 precisely because it is now the #1 risk.
2. **Combiner MSL fidelity long-tail** ‚Äî the port's *diverged*, non-stock generator (N64 3-point filter, tile-mask, `noperspective`, in-shader clamp) has no upstream template; long-tail visual drift is where Phase-2 time goes (~70% of project risk).
3. **`read_framebuffer_rgb` + probe calibration** ‚Äî the validation apparatus rides on it; a subtly wrong blit/format/flip makes gates pass/fail spuriously, and the predicted-vs-actual blend probes are calibrated to GL's UNORM rounding (needs the ¬±1-LSB tolerance or bit-exact RDP math).
4. **Pipeline-state-cache architecture** ‚Äî Metal baking blend/depth/format/MSAA/**colorWriteMask**/vertex-layout into immutable PSOs forces the deferred-state re-expression; get the key wrong (especially the easily-forgotten `colorWriteMask`) and you either explode PSO count or render color-masked passes with stale state.
5. **Tolerance-based parity is judgment, not identity** ‚Äî Metal's rasterizer/blend/filter rounding differs from GL, and the terminal Bayer-dither+RGB555 stage makes naive pixel-diff pathological; parity must be defined per-golden with dither/quantize normalized and geometry-placement metrics as the gross gate. Defining the golden set and per-frame `--max-changed-pct` budget is real work.

**Go/No-Go checkpoint ‚Äî end of Phase 2 (shader translation):** GO if the curated combiner test scene ‚Äî including the divergent features (N64 filter, RDP-memory bit-exact blend, `noperspective`) ‚Äî renders **within the per-golden `--max-changed-pct` budget, dither/quantize normalized** vs GL. If those features exhibit long-tail drift that can't be tamed to budget, that is the signal to **re-scope** (ship faithful-base-only and keep the remaster chain GL-exclusive) or **abort** before sinking Phases 3‚Äì5. Because Phases 3‚Äì5 are the bulk of the schedule and all depend on the shader translator being right, this is the correct single decision gate ‚Äî **and it is now backstopped by the Phase-1 spikes**, which independently de-risk the two non-shader unknowns (tri-count invariance and the RDP/XLU encoder-break) that a passing combiner test does *not* cover. Ahead of all of it sits the **¬ß0 pre-flight**, which can moot the entire project if SSAO can be unblocked on plain GL.

## 5. MoltenVK / Vulkan Alternative ‚Äî and Why Native Metal

**The alternative:** author a new Vulkan Fast3D backend and ship it over **MoltenVK** (Vulkan‚ÜíMetal translation) on macOS.

**Why not (rejected):**
- **It reintroduces the exact risk we're escaping.** The whole motivation is that Apple's deprecated GL-over-Metal translator op-hangs on SSAO depth-texture sampling. MoltenVK is *itself* a translation layer (Vulkan‚ÜíMetal) ‚Äî swapping one translator for another keeps the class of driver-translation risk. Native Metal removes **all** translation. This is the primary reason.
- **No template, more code.** There is **no Vulkan backend anywhere in the Fast3D/libultraship lineage** ‚Äî you'd write it from scratch, and Vulkan is far more verbose than Metal (explicit descriptor sets, pipeline layouts, render passes, memory allocation, barriers/sync) *plus* vendor MoltenVK. Native Metal has a proven ~1296-line drop-in-shaped template (`Kenix3/libultraship gfx_metal.cpp`) over this exact `CCFeatures`/vtable lineage. Estimated ~2‚Äì3√ó the effort at lower reliability.
- **Ecosystem confirms the direction.** Every mature emulator in this space (SoH/libultraship, Dolphin, PCSX2) landed on **native Metal** for macOS.
- **The only theoretical upside is moot.** A Vulkan backend would also help Linux ‚Äî but Linux GL already works fine; there is no problem to solve there.

*(Footnote, not a decision driver: Dolphin measures native Metal ~+23‚Äì30% over Vulkan-over-MoltenVK in worst cases. This is a fidelity / unblock-SSAO project, not a perf project ‚Äî perf already shipped at 101‚Äì189 fps ‚Äî so performance is **not** a go/no-go input and is cited only to note MoltenVK adds overhead, not as a reason to choose Metal.)*

**Verdict:** **native Metal.** Highest reliability (proven in this exact lineage; removes the op-hang entirely rather than trading translators), bounded effort via the libultraship template, and it satisfies the rails cleanly ‚Äî GL stays default and byte-identical everywhere, Metal is `if(APPLE)`-guarded + runtime-opt-in behind the single `gfx_rapi` switch, and gameplay-invariance holds **provided** Metal init sets `g_depth_clamp_enabled` and cross-backend equality is checked with `compare_state.py` (not asserted "by construction"). **Conditioned on ¬ß0:** if the color-encoded-depth GL dodge unblocks Phase-1 lighting in days, prefer that and shelve this backend as a future fidelity/robustness investment rather than a required unblock.
