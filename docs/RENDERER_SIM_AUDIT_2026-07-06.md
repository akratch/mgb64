## Fast3D interpreter (gfx_pc.c)

### 1. 🔴 ▫️ Unclamped current_num_lights (G_MW_NUMLIGHT) drives an OOB read+write in the lighting loop

- **File:** `src/platform/fast3d/gfx_pc.c:20111` · `gfx_sp_moveword / gfx_sp_vertex`
- **Severity:** high · **Class:** port-defect · **Category:** memory-safety · **Finder confidence:** high
- **What's wrong:** G_MW_NUMLIGHT stores the light count straight from the DL word with no clamp: `rsp.current_num_lights = ((uint32_t)data - 0x80000000U) / 32;` (line 20111). `data` is the raw command word `cmd->words.w1` (call sites 22160 PC path, 23415 N64 path), so a corrupt/mis-resolved DL, or any numlights payload below 0x80000000 (the subtraction underflows to a huge value before the uint8_t truncation), yields an arbitrary 0-255 count. The lit-vertex path in gfx_sp_vertex then iterates that count over arrays sized for only MAX_LIGHTS(=2): `for (int li = 0; li < rsp.current_num_lights - 1; li++) calculate_normal_dir(&rsp.current_lights[li], rsp.current_lights_coeffs[li]);` (16947-16948) and the shading loop at 16959-16968, plus `rsp.current_lights[rsp.current_num_lights - 1]` at 16955. current_lights is [MAX_LIGHTS+1]=3 and current_lights_coeffs is [MAX_LIGHTS]=2 (decl 3416/3418). calculate_normal_dir WRITES its 2nd argument (15460-15461 via gfx_transposed_matrix_mul/gfx_normalize_vector), so any li>=2 is a 12-byte OOB WRITE into adjacent rsp state, and current_lights[li] for li>=3 is an OOB read. The safe bound MAX_LIGHTS+1 is already known-untrusted elsewhere (line 10646 guards `current_num_lights <= (MAX_LIGHTS + 1)`), but the hot vertex path has no such guard.
- **Evidence:**

  ```
  20111: `rsp.current_num_lights = ((uint32_t)data - 0x80000000U) / 32;` (no clamp). 3418: `float current_lights_coeffs[MAX_LIGHTS][3];` with `#define MAX_LIGHTS 2` (line 210). 16948: `calculate_normal_dir(&rsp.current_lights[li], rsp.current_lights_coeffs[li]);` inside `for (int li = 0; li < rsp.current_num_lights - 1; li++)`. 15460-15461 calculate_normal_dir writes coeffs. Contrast guard at 10646: `rsp.current_num_lights <= (MAX_LIGHTS + 1)`.
  ```
- **Player impact:** A malformed/mis-decoded numlights command followed by any lit geometry corrupts neighbouring renderer state (or crashes) via an out-of-bounds write into the rsp struct; manifests as flicker/garbage lighting or a hard crash rather than a skipped draw.
- **Suggested fix:** Clamp at the set site: `rsp.current_num_lights = min(((uint32_t)data - 0x80000000U) / 32, MAX_LIGHTS + 1);` (and reject values < 0x80000000). Also bound the loop counters in gfx_sp_vertex to MAX_LIGHTS, matching the existing guard at line 10646.

### 2. 🔴 ❌ modelview_matrix_stack_size can underflow to 0, making [size-1] indexing OOB (index 255 / -1)

- **File:** `src/platform/fast3d/gfx_pc.c:16610` · `gfx_sp_pop_matrix / gfx_sp_vertex / gfx_sp_matrix`
- **Severity:** high · **Class:** port-defect · **Category:** memory-safety · **Finder confidence:** high
- **What's wrong:** gfx_sp_pop_matrix guards only against wrapping the counter, allowing it to legitimately reach 0: `if (rsp.modelview_matrix_stack_size > 0) { --rsp.modelview_matrix_stack_size; ... }` (16128). Downstream code then unconditionally indexes `[modelview_matrix_stack_size - 1]` assuming >=1. In gfx_sp_vertex the index is first narrowed to uint8_t: `uint8_t mtx_stack_pos = rsp.modelview_matrix_stack_size - 1;` (16610) -> 255 when size==0, so `rsp.modelview_matrix_stack[255]` / `modelview_cmd_addr[255]` etc. read ~14KB past the 11-entry arrays. In gfx_sp_matrix the modelview LOAD path does `memcpy(rsp.modelview_matrix_stack[rsp.modelview_matrix_stack_size - 1], matrix, sizeof(matrix));` (16025) where the int-promoted `0 - 1` = -1 -> a 64-byte OOB WRITE below the array; calculate_normal_dir (15460) also reads [size-1]. Reachable two ways: (a) an unbalanced G_POPMTX (the N64/PC dispatchers call gfx_sp_pop_matrix(1) at 22144/23005) with more pops than pushes; (b) the silent push-drop at line 16005 (`(parameters & G_MTX_PUSH) && rsp.modelview_matrix_stack_size < 11` -> push dropped when full) which unbalances pushes vs pops so later pops reach 0. Other accessors DO guard (`size > 0` at 3446, ternary at 10954), confirming the invariant is known-fragile but unenforced on the hot paths.
- **Evidence:**

  ```
  16128: `if (rsp.modelview_matrix_stack_size > 0) { --rsp.modelview_matrix_stack_size;` (allows 0). 16610: `uint8_t mtx_stack_pos = rsp.modelview_matrix_stack_size - 1;` then 16615 `mv = rsp.modelview_matrix_stack[mtx_stack_pos];` (array `modelview_matrix_stack[11]`, decl 3402). 16025: `memcpy(rsp.modelview_matrix_stack[rsp.modelview_matrix_stack_size - 1], matrix, sizeof(matrix));`. 16005: push silently dropped when `size >= 11`.
  ```
- **Player impact:** After an unbalanced matrix pop (or a scene that nests matrices past 11), the next vertex or matrix-load command reads/writes far outside the modelview stack, corrupting renderer state or crashing with SIGSEGV.
- **Suggested fix:** Never let the stack fall below 1 (clamp pop floor to 1, since index base is 1), or guard every `[modelview_matrix_stack_size - 1]` access with `size > 0`. Also account for dropped pushes at line 16005 so pop/push stay balanced (e.g., track an overflow depth counter to swallow the matching pops).
- **Verification:** REFUTED (adj. severity low, real-player-issue=False). The cited code substantially exists but the defect is not reachable with the data this port actually processes, and the cited line numbers are partly wrong.

What matches: the pop guard at 16128 is verbatim (`if (rsp.modelview_matrix_stack_size > 0) { --...; }`), permitting the counter to reach exactly 0. The unguarded `[modelview_matrix_stack_size - 1]` indexing exists (gfx_sp_vertex, the LOAD memcpy at 16025, calculate_normal_dir at 15460), and would compute index 255 (uint8_t) or -1 (int) if size were 0. Array decl is [11] at 3402.

What's wrong/unsubstantiated: (1) The cited line 16610 is NOT the vertex index — it's a diagnostic printf inside a reject-logger. The real `uint8_t mtx_stack_pos = rsp.modelview_matrix_stack_size - 1;` is at 16642. (2) Reachability is not substantiated. gfx_sp_reset (23722) initializes the stack to size=1. Both G_POPMTX dispatch sites (22144, 23005) call gfx_sp_pop_matrix(1) — always a single decrement, never a bulk count. So reaching 0 requires strictly more POPMTX commands than PUSH commands since reset, i.e. a malformed display list. This port replays fixed, authored retail GoldenEye display lists, which are well-formed and balance push/pop; the base level-1 matrix is never popped by well-formed DLs. (3) The push-drop path at 16005 only drops when the stack is already 11 deep — GoldenEye's matrix nesting (rooms/bones) is only a few levels deep and never approaches 11, so this path is effectively unreachable and cannot create the imbalance described. (4) This structure is faithful to upstream fast3d (SM64 PC port), where the identical pattern has run for years without this firing, precisely because it relies on well-formed DL input. Project history notes the engine runs across all 18+ levels without crashes.

So it's a latent, upstream-shared fragility gated behind an externally-guaranteed invariant (balanced push/pop from fixed retail data), not a demonstrated reachable defect against the inputs the port receives.
  - **Correction:** The described unguarded indexing is real but the vertex-path line is 16642, not 16610 (16610 is a diagnostic printf). More importantly, the underflow-to-0 is only reachable via a malformed display list (more POPMTX than PUSHMTX, or 11-deep nesting) — neither occurs in the fixed, well-formed retail GoldenEye display lists this port replays. Both G_POPMTX sites call pop(1), reset sets size=1, and the code is faithful to battle-tested upstream fast3d. It is a latent hardening opportunity (clamp the pop floor to 1), not a defect that affects players with the game's actual data.

### 3. 🟠 ❌ Coverage-memory XLU strips are excluded from the per-group deferred re-snapshot, forcing overlapping backdrop tris to share one stale per-batch framebuffer snapshot

- **File:** `src/platform/fast3d/gfx_pc.c:18376` · `gfx_emit_loaded_triangle`
- **Severity:** medium · **Class:** port-defect · **Category:** visual-artifact · **Finder confidence:** medium
- **What's wrong:** The RDP coverage-memory emulation reads the framebuffer 'memory' color from a snapshot taken ONCE per draw_triangles batch (gfx_opengl_draw_triangles per-batch path, gfx_opengl.c:2266-2285; mtl_draw_triangles single snapshot, gfx_metal.mm:2980-3009). The per-batch snapshot is only correct if same-material tris in the batch are coplanar/non-overlapping, since every fragment reads the pre-batch framebuffer. The room-XLU sort/defer machinery WOULD fix overlap by splitting each cmd_addr group into its own draw_triangles call (each triggering a fresh snapshot in gfx_room_xlu_deferred_draw_pending, line 13977-13983). But coverage-memory tris are structurally excluded from that path: gfx_api_blend_mode_for_rdp_memory_class sets api_blend_mode to GFX_BLEND_ALPHA_RDP_CVG_MEMORY (line 18355-18357), while room_secondary_xlu_sort requires 'api_blend_mode == GFX_BLEND_ALPHA' (line 18376). So sortable=0 for every coverage tri (line 19542-19543: buf_vbo_tri_sortable = room_secondary_xlu_sort ? 1 : 0), and gfx_room_xlu_defer_buffer bails if ANY tri is non-sortable (line 14012-14016). Result: all coverage backdrop tris land in one batch with one snapshot. Where a secondary-room backdrop shell overlaps in screen space (e.g. near+far faces of an unculled fog shell, or layered haze strips), the second-drawn tri reads pre-batch memory instead of the first tri's blended+coverage-accumulated result, so the RDP 3-bit coverage-wrap term and color are wrong (banding / under-darkened fog vs N64).
- **Evidence:**

  ```
  room_secondary_xlu_sort at line 18370-18382 requires `blend_mode == GFX_BLEND_ALPHA && api_blend_mode == GFX_BLEND_ALPHA`; api_blend_mode was just set for the coverage class at 18355: `gfx_api_blend_mode_for_rdp_memory_class(blend_mode, rdp_memory_blend_class)` -> GFX_BLEND_ALPHA_RDP_CVG_MEMORY. Emission: line 19542 `buf_vbo_tri_sortable[buf_vbo_num_tris] = room_secondary_xlu_sort ? 1 : 0;`. Defer gate: line 14012 `for (...) if (!buf_vbo_tri_sortable[i]) return false;`
  ```
- **Player impact:** On secondary-room translucent/fog backdrops whose strips overlap in screen space, the coverage-wrap blend is under-accumulated: subtly wrong tint/darkness or faint banding on those backdrops. Single-layer non-overlapping shells are unaffected.
- **Suggested fix:** Either allow coverage-memory tris into the defer/sort path (relax the api_blend_mode==GFX_BLEND_ALPHA gate to also accept GFX_BLEND_ALPHA_RDP_CVG_MEMORY so each cmd_addr group re-snapshots), or explicitly force per-tri snapshot for the coverage class when the batch's tri bboxes overlap.
- **Verification:** REFUTED (adj. severity low, real-player-issue=False). The structural claim is real: coverage-class tris get api_blend_mode=GFX_BLEND_ALPHA_RDP_CVG_MEMORY (gfx_pc.c:5116, 18387-18389), room_secondary_xlu_sort requires api_blend_mode==GFX_BLEND_ALPHA (18402-18414, NOT the cited 18376), so coverage tris are non-sortable (19580) and the defer path bails on them (14017-14021). But the finder's causal mechanism is a misread. The framebuffer snapshot exists ONLY for the memory/coverage-memory classes — gfx_opengl.c:2245 gates it on g_blend_alpha_rdp_memory||g_blend_alpha_rdp_cvg_memory. The plain GFX_BLEND_ALPHA tris that the sort/defer path actually operates on take NO snapshot and use native fixed-function per-fragment alpha blending against the live framebuffer, correct regardless of batching. So the sort/defer machinery is a back-to-front depth sort, not the 'per-group re-snapshot overlap fix' the finder describes; coverage tris being excluded from it does not remove any snapshot they would otherwise get. The genuine intra-batch overlap approximation for CVG_MEMORY is explicitly documented at gfx_opengl.c:2266-2274 AND already has an opt-in exact path, GFX_XLU_SNAPSHOT_PER_TRI / GE007_XLU_SNAPSHOT_MODE=pertri (2248-2264), that re-snapshots per triangle and is 'Correct for intra-batch overlap.' The suggested fix (route coverage tris into the sort path) would not produce per-tri snapshots and so would not fix intra-group overlap anyway. Additionally the one known secondary-room coverage-shell case (Frigate, gfx_room_water_alpha_suppress_needed at 5125+) has its alpha suppressed so it does not visibly shade the framebuffer. No evidence of an actual observed artifact. This is a documented approximation with an existing correct-path hatch, not an unhandled defect.
  - **Correction:** Coverage-class secondary-room XLU tris are indeed excluded from the room-XLU sort/defer gate (real code at gfx_pc.c:18402-18414, 19580, 14017-14021; the finder's line numbers are ~26 lines low). However, that path is a depth-ordering sort for plain fixed-function alpha blending, not a per-group framebuffer-snapshot mechanism — plain GFX_BLEND_ALPHA tris take no snapshot (gfx_opengl.c:2245). The real intra-batch overlap approximation applies only to the CVG_MEMORY class and is already a documented performance tradeoff (gfx_opengl.c:2266-2274) with an existing exact opt-in path GFX_XLU_SNAPSHOT_PER_TRI/GE007_XLU_SNAPSHOT_MODE=pertri (2248-2264). ### 4. 🟠 ✅ 64-bit pointer hash table keys on low 32 bits only — aliasing returns the wrong buffer and window-overflow silently drops mappings

- **File:** `src/platform/gfx_ptr.h:29` · `gfx_ptr_store / gfx_ptr_resolve`
- **Severity:** medium · **Class:** port-defect · **Category:** correctness · **Finder confidence:** medium
- **What's wrong:** The table that recovers full 64-bit pointers from GBI-truncated 32-bit display-list addresses is keyed purely on the low 32 bits of the pointer (`uint32_t key = (uint32_t)full;`). Two distinct live allocations whose addresses differ only in the high 32 bits (i.e. 4GB apart — plausible with a large malloc'd ROM buffer plus mmap'd texture-pack pages plus the gfx arena) collapse to the same key; the second gfx_ptr_store() overwrites the first (key-match branch), so gfx_ptr_resolve() then returns the WRONG buffer for whichever DL referenced the evicted one — reading foreign/short data as vertices/matrices. Separately, the store uses a fixed 4-slot linear-probe window and, when all four slots hold *other* keys, unconditionally evicts base_idx (`gfx_ptr_keys[base_idx] = key;`). The evicted mapping is lost: a later gfx_ptr_resolve() of the evicted key walks the 4 slots, matches none, and returns NULL. Because the table is only cleared at renderer init (gfx_pc.c:23775 memset), not per frame, distinct keys accumulate across a whole level, and in a dense scene some 4-window will overflow (birthday effect over the effective bucket count) → dropped draw.
- **Evidence:**

  ```
  gfx_ptr.h:31 `uint32_t key = (uint32_t)full;`  gfx_ptr.h:37-41 store overwrites on key match `if (gfx_ptr_keys[idx] == key || gfx_ptr_keys[idx] == 0) { gfx_ptr_keys[idx] = key; gfx_ptr_vals[idx] = full; return; }`  gfx_ptr.h:43-45 `/* All probe slots occupied — evict first slot */ gfx_ptr_keys[base_idx] = key; gfx_ptr_vals[base_idx] = full;`  gfx_ptr.h:51-57 resolve returns NULL once the 4-probe window is exhausted. Populated via platform_os.h:513 `gfx_ptr_store(addr)` in osVirtualToPhysical_pc.
  ```
- **Player impact:** Intermittent wrong/missing geometry (a mesh drawn with another object's vertices, or a draw silently skipped) in heavy scenes or long sessions, hard to reproduce and not tied to any single asset.
- **Suggested fix:** Store the high bits too (compare full uintptr_t, not just the 32-bit key) so an aliased low-32 key can't shadow a live buffer, and on window overflow grow/relocate rather than evicting a possibly-live mapping (or at least log + fail closed). Given the 64K-slot table this is cheap.
- **Verification:** CONFIRMED (adj. severity low, real-player-issue=True). The cited code matches the evidence verbatim (gfx_ptr.h:31,37-45,66-76) and the table is cleared only at init (gfx_pc.c:23775), so distinct keys do accumulate per level. The eviction sub-claim is a genuine defect: on a full 4-probe window, gfx_ptr_store overwrites base_idx with no tombstone, and a later gfx_ptr_resolve of the evicted key walks all 4 slots, matches none, encounters no zero slot, and returns NULL — a silently dropped mapping. gfx_resolve_addr then returns NULL and callers null-check, so the outcome is a dropped draw, not corruption or a crash. However the headline aliasing mechanism is mischaracterized: resolve is fundamentally handed only a 32-bit token (GBI truncates pointers to 32-bit; gfx_ptr_resolve(uint32_t key) / gfx_resolve_addr(uint32_t addr)), so two live buffers sharing identical low-32 bits are indistinguishable at resolve time no matter what is stored. The finder's suggested fix ('store the high bits, compare full uintptr_t') cannot work for resolution because there are no high bits available in the resolve key — this is an inherent limitation of the port's truncation scheme, not a fixable keying bug, and it requires two live allocations with byte-identical low-32 bits (very low probability). So the real, code-substantiated issue is the silent no-tombstone eviction on window overflow, a low-probability dropped-draw robustness gap.
  - **Correction:** The actionable defect is the silent eviction on a full 4-slot probe window (gfx_ptr.h:43-45): it drops a possibly-live mapping without a tombstone, so a later resolve returns NULL and the draw is skipped. Impact is an intermittent dropped draw (callers null-check gfx_resolve_addr), not a wrong-buffer read or crash, and it is low-probability given the 65536-slot table. The finding's primary 'low-32 aliasing returns the wrong buffer' framing and its suggested fix are incorrect: gfx_ptr_resolve only ever receives a 32-bit truncated token, so storing/comparing the high 32 bits cannot disambiguate aliased pointers — low-32 keying is inherent to the GBI truncation, not a fixable table defect. Severity is low, not medium.

### 5. 🟠 ✅ N64 display-list recursion has no depth cap (unlike the PC path) -> C-stack overflow on cyclic/deep DL

- **File:** `src/platform/fast3d/gfx_pc.c:22705` · `gfx_process_n64_dl`
- **Severity:** medium · **Class:** port-defect · **Category:** robustness · **Finder confidence:** high
- **What's wrong:** gfx_process_n64_dl handles a push-type G_DL by recursing into itself via native C recursion: `gfx_process_n64_dl((const uint8_t *)dl_addr);` (23267). It increments the shared dl_depth (22705) but never checks it against any maximum. The sibling PC interpreter gfx_run_dl_pc DOES cap recursion: `dl_depth++; if (dl_depth > 32) { ... dl_depth--; return; }` (22028-22033). The only N64-side bounds are the per-invocation 50000-command counter (22761) and the g_n64_dl_stack[128] write guard (22707) — neither limits recursion DEPTH. A corrupt or mis-resolved N64 DL that forms a cycle (A->B->A push-type) or nests very deeply passes the plausibility/segment checks and recurses until the native stack overflows (SIGSEGV), instead of the intended graceful skip. gfx_process_n64_dl frames are large (many locals + fprintf temporaries), so the overflow arrives quickly.
- **Evidence:**

  ```
  22705: `dl_depth++;` with no subsequent `if (dl_depth > N)` in gfx_process_n64_dl. 23267: `gfx_process_n64_dl((const uint8_t *)dl_addr);` (recursion). Contrast gfx_run_dl_pc 22029: `if (dl_depth > 32) { g_effect_pending_child_label = NULL; dl_depth--; return; }`.
  ```
- **Player impact:** A corrupt asset or a display-list pointer that mis-resolves into a self-referential region crashes the process via stack overflow rather than being skipped like every other malformed-DL case the interpreter defends against.
- **Suggested fix:** Add the same depth guard as gfx_run_dl_pc at the top of gfx_process_n64_dl (e.g. `if (dl_depth > 32) { ...restore state...; return; }`) before the recursive descent, reusing the ABORT_N64_DL cleanup macro.
- **Verification:** CONFIRMED (adj. severity medium, real-player-issue=True). Re-read gfx_process_n64_dl and gfx_run_dl_pc in src/platform/fast3d/gfx_pc.c. The asymmetry is real: gfx_run_dl_pc caps recursion at 22066-22071 (dl_depth++; if (dl_depth > 32) { dl_depth--; return; }), but gfx_process_n64_dl increments the shared dl_depth at 22743 with no depth check anywhere before recursing into itself at 23305 for a push-type G_DL. I confirmed the only N64-side bounds are per-invocation and non-depth: n64_cmd_count is reset to 0 at each entry (22701) and only trips at 22799 (>50000) within a single invocation, so it cannot limit recursion depth; the g_n64_dl_stack write is guarded at 22745 only to avoid an OOB array store, not to stop recursion (depth continues past the 128-entry array). A corrupt/mis-resolved DL forming a cycle (or deep nest) that passes gfx_addr_is_n64_data + gfx_n64_dl_start_is_plausible would recurse on the native C stack until SIGSEGV, whereas the function is otherwise heavily instrumented to skip malformed DLs gracefully. The finder's cited line numbers are each off by a few lines (22705 is 'int entry_depth = dl_depth;' not the increment; 23267 is a boot-logo fprintf not the recursion at 23305; 22029 is seg_addr not the PC cap at 22067), but every cited construct exists nearby and the described defect is accurate and verifiable.
  - **Correction:** Line numbers are imprecise: the uncapped increment is at gfx_pc.c:22743 (not 22705); the self-recursion is at 23305 (not 23267); the sibling PC-path cap is at 22066-22071 (not 22029-22033); the per-invocation 50000-command counter is at 22799 (not 22761) and is reset per call so it cannot bound recursion depth. Substance of the finding — N64 DL recursion lacks the depth cap the PC path has, so a cyclic/deep DL overflows the native stack instead of being skipped — is correct.
